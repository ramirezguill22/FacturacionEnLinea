====================================
BLOQUE 5. COMO FUNCIONA AHORA EL FLUJO
====================================

El flujo actual funciona asi:

1. El usuario captura un ticket en la pantalla del portal.

2. El frontend no llama directamente a NetSuite.

3. El frontend envia la peticion a su ruta interna de Next.js:
- /api/tickets/validar

4. Esa ruta interna usa la capa compartida del frontend para reenviar la peticion al backend real.

5. La base URL del backend ya no esta escrita en cada pantalla.
Se toma desde la variable:
- BACKEND_API_BASE_URL

6. En ambiente local de frontend, BACKEND_API_BASE_URL puede apuntar al backend desplegado en Cloud Run.

7. El backend desplegado recibe el ticket, valida el request, consulta NetSuite mediante el RESTlet y devuelve una respuesta normalizada.

8. El frontend recibe esa respuesta y muestra el resultado en la pantalla de validacion.

====================================
BLOQUE 6. VARIABLES Y RESPONSABILIDADES
====================================

Frontend:
- usa BACKEND_API_BASE_URL para saber a que backend debe llamar

Backend:
- usa sus propias variables de entorno para conectarse a NetSuite
- sigue necesitando:
  - NETSUITE_RESTLET_URL
  - NETSUITE_ACCOUNT
  - NETSUITE_REALM
  - NETSUITE_CONSUMER_KEY
  - NETSUITE_CONSUMER_SECRET
  - NETSUITE_TOKEN_ID
  - NETSUITE_TOKEN_SECRET

Regla importante:
- el frontend nunca debe tener las credenciales de NetSuite
- solo el backend conoce y usa esos valores

BLOQUE 8. CONFIGURACION LOCAL DEL FRONTEND
====================================

Para probar el frontend local contra el backend desplegado se uso un archivo local no versionado:
- .env.local

Con este valor:

Esto permite que el portal en localhost use el backend real desplegado.
